## 评价约束

所有评语和改进建议必须**绑定原回答的具体内容**，禁止使用不引用原回答的空泛评价。

### 禁止 vs 必须

| 禁止（空泛套话） | 必须（绑定原回答） |
|----------------|------------------|
| "回答不够深入" | "你说项目中用了Redis，但没解释为什么选Redis而不是本地缓存——追问一下具体场景会更有说服力" |
| "逻辑可以更清晰" | "你讲了技术选型又跳到团队分工再跳回技术，建议按'问题→方案→结果'三段式重组" |
| "表达可以更流畅" | "你把该回答的判断说成了'可能会导致最终信息不一致'——太书面了，口语直接说'数据对不上'就行" |
| "整体不错，有提升空间" | 禁止使用。要么说清楚哪里好哪里差，要么不写 |
| "建议多准备" | "这道题问的是微服务拆分原则，你答的是部署流程——跑题了，应该从'按业务边界拆'和'数据独立性'切入" |

### 改进建议三要素

每条改进建议必须同时包含：

1. **定位**：你的回答在哪个点上出了问题（跑题/缺细节/前后矛盾/表述模糊）
2. **对照**：和原问题要求差在哪里（问题问的是A，你答的是B）
3. **示范**：给出一个具体的修正方向或替换措辞

示例：

> 问题问的是"项目里最大的技术难点怎么解决的"，你答的是项目整体功能介绍。
> - **定位**：跑题，问题要的是困难和解决过程
> - **对照**：你花80%篇幅介绍项目做什么，只一句"遇到性能问题优化了一下"带过
> - **示范**：建议选一个具体难点按"现象→排查→方案→效果"四步展开，比如"首页加载8秒→查Network发现API串行调用→改为并行+缓存→降到1.2秒"

### 一致性检查（生成评价后自检）

| 检查项 | 规则 |
|--------|------|
| 评价是否引用了用户回答中的原句？ | 每条至少引用一处 |
| 改进建议是否指向了该题特有的问题？ | 不能是通用建议套到任何题上 |
| 得分是否和评语一致？ | 如果写的全是问题还给80分，自相矛盾 |
| 评价是否与简历信息冲突？ | 如用户简历写"3年Java"，不能说"你没Java经验" |
